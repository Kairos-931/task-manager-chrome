# PRD：TaskMaster - Chrome 任务管理插件

> 最后更新：2026-06-04 | 当前版本 v1.2.0（Chrome 扩展）+ v3.2.0（含手机同步）

## 1. 产品概述

TaskMaster 是一款功能完整的 Chrome 浏览器任务管理插件，旨在帮助用户高效管理日常任务。通过直观的多视图界面和强大的同步功能，用户可以在任何设备上无缝管理个人和工作任务。

- **解决问题**：用户需要一个集成在浏览器中的任务管理工具，支持多设备同步，提供多种视图模式，帮助用户更好地规划和跟踪任务进度。不同于 Todoist、滴答清单等独立应用（需要切换窗口、打开 App），TaskMaster 住在浏览器工具栏里，点击即用、关上即走。
- **目标用户**：需要管理个人和工作任务的浏览器用户，特别是需要跨设备同步任务的重度浏览器用户和知识工作者。
- **产品价值**：最低摩擦力的任务管理——无需切换应用，浏览器内一键触达。通过简化任务管理流程，提高工作效率和时间管理能力，减少任务遗漏和拖延。
- **差异化**：① 不是独立 App，是浏览器原生扩展，零上下文切换 ② 支持手机通过 Telegram/网页快速录入，桌面 Chrome 深度管理 ③ 四种视图（列表/日/周/月）在弹窗内完成，比同类插件功能更完整

## 3. 核心流程

### 3.1 任务管理流程

1. **创建任务**：用户点击"添加"按钮，填写任务标题、描述、优先级、分类、截止日期、预计时长、重复规则等信息，点击保存。
2. **编辑任务**：用户点击任务旁边的编辑按钮，修改任务信息，点击保存。
3. **完成任务**：用户点击任务旁边的 checkbox，标记任务为完成状态。
4. **删除任务**：用户点击任务旁边的删除按钮，确认后删除任务。
5. **任务过滤**：用户通过筛选器选择不同的过滤条件（优先级、分类、隐藏已完成/今日之前），查看符合条件的任务。
6. **视图切换**：用户点击视图切换按钮，在列表、日、周、月视图之间切换。
7. **拖拽调整**：用户在日/周/月视图中拖拽任务到不同日期。

### 3.2 数据同步流程

1. **自动同步**：当任务数据发生变化时，系统自动将数据同步到 Chrome 云存储（chrome.storage.sync）。
2. **双重存储**：每次写入 sync 时同时写入 local 作为备份，防止 sync 被清空导致数据丢失。
3. **数据合并**：当检测到远程数据更新时，系统按 updatedAt 时间戳自动合并，两端修改不丢失。
4. **手动同步**：用户可通过同步管理面板手动上传云端、拉取云端、导出文件、导入文件。
5. **本地恢复**：检测到 sync 数据被清空时，自动从 local 备份恢复。

### 3.3 手机端录入流程

**场景A：手机网页**

> 打开 PWA（添加到主屏幕）→ 填写任务名 → 选优先级/分类/日期 → 提交 → 任务写入 Worker pending_tasks 表 → Chrome 插件打开时自动拉取 → 合并到本地任务列表 → 标记已同步

**场景B：Telegram Bot**

> 发消息 "明天 高 完成报告" → Bot 解析：日期=明天，优先级=高，标题=完成报告 → 写入 pending_tasks → Chrome 插件自动拉取

**支持的自然语言格式：**
- "买牛奶" → 标题，日期=今天
- "明天 高 完成报告" → 高优先级，明天
- "#工作 后天 低 准备演示" → 分类=工作，低优先级，后天
- "3月15日 中 开会" → 具体日期
- "周三 高 写周报" → 星期几

## 5. 技术方案

### 5.1 技术栈

| 层 | 技术 | 选择理由 |
| :--- | :--- | :--- |
| **开发语言** | TypeScript | 类型安全，减少运行时 bug |
| **扩展框架** | Chrome Extension MV3 | Chrome 官方最新标准，Service Worker 替代后台页 |
| **样式** | Tailwind CSS | 原子化 CSS，快速迭代 UI |
| **打包** | esbuild (IIFE) | 极快构建速度，输出单文件，适合扩展场景 |
| **本地存储** | chrome.storage.sync + local | Chrome 原生 API，零依赖跨设备同步 |
| **后端** | Cloudflare Worker + D1 | 免费额度足够个人使用，全球边缘部署 |
| **手机端** | 内嵌 HTML（Worker 直接 serve） | 一个 Worker 搞定 API + 页面 + Telegram Bot |
| **Telegram** | Telegram Bot API | webhook 模式，自然语言解析添加任务 |

### 5.2 关键技术实现

**1. 数据存储**

- 使用 chrome.storage.sync 进行跨设备数据同步，总容量约 100KB
- 使用 chrome.storage.local 进行本地备份，双写确保数据安全
- 数据分块存储（tm_meta + tm_tasks_0, tm_tasks_1, ...）突破单条 8KB 限制
- 三级回退读取：sync 分块 → 旧版 simple key → local 备份
- 旧版数据自动迁移，升级无感知
- 空数据保护：loadState 返回空数据时跳过 persistState，防止覆盖真实数据

**2. 数据同步**

- 监听 chrome.storage.onChanged 事件，自动合并远程数据
- 按 updatedAt 时间戳解决冲突，两端修改不丢失
- 提供手动同步功能（四宫格面板），允许用户主动触发
- 手机端同步：Chrome 插件打开时自动拉取 Worker pending_tasks，合并后标记已同步

**3. 视图渲染**

- TypeScript 编译后通过 esbuild 打包为单文件 IIFE
- 使用模板字符串生成 HTML 内容
- 支持列表、日、周、月四种视图模式
- 共用一套渲染逻辑，弹窗和全屏通过容器尺寸区分

**4. 任务管理**

- 支持任务的增删改查操作
- 实现任务拖拽功能，支持在不同日期之间移动任务
- 支持任务重复规则设置（每日/每周/每月/工作日/自定义）
- 模块化代码结构：types.ts、storage.ts、task.ts、render.ts、events.ts、entry.ts

**5. 性能优化**

- 实现数据缓存，减少存储操作次数
- 使用事件委托减少事件监听器数量
- 优化渲染性能，避免不必要的 DOM 操作

Task {
  id: string              // 任务唯一标识（UUID）

## 7. 功能边界

1. **存储限制**：chrome.storage.sync 总容量约 100KB，单条 8KB，通过分块存储解决
2. **同步条件**：需要登录同一 Google Chrome 账号并开启同步；国内需要 VPN
3. **离线功能**：支持离线操作，数据先存本地，网络恢复后自动同步
4. **浏览器兼容性**：仅支持 Chrome 浏览器
5. **数据安全**：Chrome 端数据存储在用户的 Chrome 账户中；手机端数据存储在 Cloudflare D1 数据库
6. **workers.dev 限制**：Cloudflare Worker 域名国内需要 VPN，后续考虑绑定自定义域名

## 9. 部署与发布

### 9.1 架构概览

**三层架构：**

1. **Chrome Extension（前端）** — popup + newtab，通过 chrome.storage.sync/local 管理数据
2. **Cloudflare Worker + D1（后端）** — 处理手机端任务录入、Telegram Bot webhook、分类同步
3. **手机端入口** — 手机网页（PWA 表单）+ Telegram Bot

**数据流向：**
- Chrome 端 ↔ Chrome 云存储（跨设备同步）
- 手机端 → Worker → D1 数据库
- Chrome 端定时拉取 D1 pending_tasks → 合并到本地

**后端地址：** `taskmaster-api.yx9391.workers.dev`

### 9.2 部署流程

**Chrome 扩展：**
1. 本地构建：`npm run build`（TypeScript 编译 → Tailwind 构建 → esbuild 打包 → 复制资源）
2. 加载到 Chrome：`chrome://extensions/` → 开发者模式 → 加载已解压扩展 → 选择 `chrome-extension-sync/`
3. 更新：重新构建后在 `chrome://extensions/` 点击刷新按钮

**后端 Worker：**
1. 代码在 `backend/` 目录
2. 通过 `wrangler deploy` 部署到 Cloudflare Workers
3. D1 数据库通过 `wrangler d1` 管理

### 9.3 版本管理

- 版本号格式：主版本.次版本.修订版本（如 1.2.0）
- 更新策略：
    - **主版本更新**：重大功能变更或架构调整
    - **次版本更新**：新功能添加或较大改进
    - **修订版本更新**：bug 修复或小的改进
