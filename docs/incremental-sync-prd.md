# 增量同步改造 PRD

## 目标

让 TaskMaster 的本地数据与云端数据在设备联网后自动收敛，避免全量快照覆盖导致另一台设备的任务、完成状态或删除操作被静默丢失。

## 用户与核心体验

用户仍然只需正常新增、编辑、完成或删除任务。离线时继续使用本地数据；恢复联网后系统自动同步，不弹出“保留本机还是云端”的选择。

## 核心功能

- 客户端把任务、分类和设置的新增、修改、删除保存为可重试的变更队列。
- Worker 为每条变更分配递增的服务端游标，并返回上次游标之后的变更。
- 删除操作保留为墓碑记录，不让旧设备重新上传已删除任务。
- 同一任务并发修改时，按修改时间自动取较新的版本；不同任务的修改都会保留。
- 第一次使用新协议时，Worker 自动把旧版 `full_sync` 快照迁移为增量记录。

## 视觉与交互

不增加冲突弹窗。沿用现有同步状态和同步面板；手动同步仍可触发一次立即收敛，但不会覆盖另一端未同步的任务。

## 约束

- 扩展端源码：`shared/*.ts`，构建后复制到 `chrome-extension-sync/`。
- Worker 与 D1：`backend/index.js` 和 `backend/schema.sql`，部署 Worker 前需执行 schema。
- 保留旧版 `/api/fullsync`，让尚未升级的扩展不立即失效。
- 删除墓碑不自动清理，优先保证旧设备恢复联网后不会复活任务。

## 验收标准

1. 两台设备离线各新增任务，联网后两条任务都存在。
2. 一台删除任务，另一台旧设备同步后不会复活该任务。
3. 两台设备修改不同任务，联网后两处修改都存在。
4. 同一任务并发编辑时自动采用较新修改，不出现整库覆盖。
5. 旧版 `full_sync` 数据首次访问新接口后可恢复为增量记录。

## 竞品参考

采用 Todoist、Notion 等离线优先产品常见的“本地缓存 + 变更日志 + 服务端游标”模型，而非同步整份文档。
