# 变更记录规范

## 为什么要记录

提交只能说明“代码发生了什么”，不能说明“为什么做、用户受什么影响、是否已验收”。Issue 是每一项独立产品或工程变更的长期记录；Release 只是一个版本的汇总，不替代 Issue。

## 必须建 Issue 的变更

- 新功能、交互或用户流程变化。
- 用户可感知的 bug 修复。
- 数据模型、同步协议、权限或部署行为变化。
- 测试、质量门禁、构建与发布机制等工程保障变化。

同一用户目标内的小改动可放入一个 Issue；能独立说明目标、影响和验收的内容必须拆分为不同 Issue。

## Issue 最低内容

1. 问题或目标。
2. 用户影响和涉及端。
3. 计划范围与验收标准。
4. 交付后补充验证命令、提交、版本标签和 Release 链接。

## 发布闭环

1. 实现前建立或关联 Issue。
2. 提交信息或 PR 描述引用 Issue。
3. 发版时建立 Git tag 和 GitHub Release。
4. Release 汇总各变更 Issue；各已完成 Issue 回写版本并关闭。
5. 未完成的拆分项保持开放，不混入已完成发布 Issue。
